# Result Folder Schema

Each Chern-degree folder is a verified or pending milestone record, not a raw
artifact dump.

The root [`summary.json`](../summary.json) mirrors the status table in
machine-readable form.  It is intended for quick inspection and for the
lightweight validator; the authoritative per-degree proof objects remain the
degree-folder certificates.

## Required Fields

`README.md` should record:

- Chern degree.
- Status: `pending`, `theorem-assisted candidate`, `verified`,
  `verified relation` for a full relation certificate, or `needs rerun`.
- Rank and nullity.
- Source row dimension and W-basis dimension.
- Selected row names and selected W-column names.
- Exact manifest, worker, reduce, and verification commands.
- Primary prime and optional second prime.
- Elapsed wall time for manifest, workers, reduce, and verification.
- Hashes of the manifest, reduce output, verification output, and source files.
- Hashes or labels for the raw run ledger/provenance, without committing those
  raw files.
- Computed W-column count and whether the full W basis is covered.

`certificate.json` should record only compact certificate data:

Raw shards, caches, complete matrices, and long logs should remain outside this
repository. Prefer generating `certificate.json` with
`scripts/extract_verified_result.py` rather than editing it by hand.

The extractor rejects non-final reduce artifacts by default. Do not publish
from an intermediate `--allow-in-progress` verification artifact.

Run the public-record validator after adding or editing committed artifacts:

```bash
python scripts/validate_publication.py --all
```

## Computed Column Exports

`computed_columns_mod_p.json.gz` records the pairing columns actually computed
for a verified certificate.  It has scope `computed_shards_only`; readers
should not treat it as a full matrix unless `full_w_basis_covered` is true.

The export is generated by `scripts/export_computed_columns.py` and includes:

- primary prime;
- source and W basis digests;
- manifest/reduce/verification hashes;
- one vector per computed W-column;
- a flag saying whether each column was used in the selected-minor certificate.

## Theorem-Assisted c12 Candidate

For the current `c12` publication state, the expected file is
`theorem_assisted_candidate.json`.  It records:

- source dimension `44`;
- selected rank `43` and selected nullity `1`;
- selected rows and columns;
- nonzero selected minor determinant modulo the primary prime;
- modular kernel vector;
- checked loaded-column count;
- rational reconstruction status and primitive integer coefficients;
- explicit theorem assumptions required for the conclusion;
- source, manifest, basis, and postprocessor hashes.

This artifact identifies the candidate relation line under the external
uniqueness theorem.  It is not a full `1039`-column annihilation certificate.

## Full Relation Certificates

For c12, the expected file is `relation_certificate.json`, generated by
`scripts/extract_relation_certificate.py`.  It records:

- rank 43 and nullity 1;
- selected rank-43 rows and columns;
- nonzero selected minor determinant modulo the primary prime;
- normalized left-kernel vector modulo the primary prime;
- full W-basis annihilation certificate;
- second-prime selected-minor and annihilation check;
- exact commands and provenance hashes.

The modular kernel vector is not an exact rational relation by itself.  A
publication of the literal rational relation should include an additional
reconstruction/exact-verification artifact.
